# 06 - المتطلبات التقنية والبيانات

## الوضع الحالي

المشروع حالياً يحتوي نماذج Prisma مهمة:

- `Client`
- `Order`
- `Payment`
- `BrandKit`
- `Deliverable`
- `Lead`
- `User`

لكن بما أن المشروع ليس نظام مالي، يجب التعامل مع `Payment` كشيء غير أساسي أو مخفي من الواجهة.

## قواعد مهمة قبل أي تعديل

- لا يتم تغيير database schema بدون موافقة صريحة.
- لا يتم الاعتماد على Render filesystem لحفظ thumbnails.
- لا يتم إظهار زر أو رابط إذا لم تكن وظيفته مكتملة.
- لا يتم إدخال منطق دفع داخل النظام.

## البيانات التي يمكن استخدامها في Phase 1 بدون تعديل schema

### Client

يمكن استخدام:

- `name`
- `email`
- `phone`
- `source`
- `orders`
- `brandKit`
- `deliverables`

### BrandKit

يمكن استخدام:

- `colors`
- `fonts`
- `styleTags`
- `notes`

### Order

يعرض في الواجهة باسم:

- طلب
- مشروع
- Request

بدل التركيز على الأسعار أو الدفع.

## API المطلوب لاحقاً

### Clients

- `GET /api/admin/clients/summary`
- `GET /api/admin/client/:id`
- `PATCH /api/admin/client/:id`

### Requests

- `POST /api/admin/client/:id/requests`
- `PATCH /api/admin/requests/:id`
- `GET /api/admin/client/:id/requests`

### Deliverables

- `POST /api/admin/requests/:id/deliverables`
- `PATCH /api/admin/deliverables/:id`
- `DELETE /api/admin/deliverables/:id`

## Storage recommendation

للتسليم الحقيقي، الأفضل استخدام storage خارجي.

الخيارات:

- Cloudinary: سهل للصور والتحويلات.
- Supabase Storage: مناسب لو بدنا نظام ملفات بسيط.
- Cloudflare R2: ممتاز للتكلفة والتحكم.

## مخاطر يجب تجنبها

- رفع الصور إلى Render local disk.
- حفظ صور thumbnails الكبيرة كـ base64 داخل قاعدة البيانات.
- فتح preview links بدون حماية أو token.
- إرسال بيانات admin كاملة إلى صفحة public preview.
- تحميل كل العملاء بكل الطلبات والملفات في dashboard الرئيسي.

## Checks قبل التنفيذ

قبل أي push مستقبلي:

- `npm run build`
- `npm run lint`
- `npx prisma validate` إذا تم لمس Prisma أو backend data flow

إذا كان هناك migration:

- لا يتم تطبيقها على Neon بدون موافقة صريحة.
