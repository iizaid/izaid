# 05 - خطة التنفيذ بالمراحل

## Phase 1 - تحسين الواجهة بدون تغيير قاعدة البيانات

هذه أول مرحلة مقترحة لأنها منخفضة المخاطر.

### الهدف

تحسين الشكل وتجربة الاستخدام اعتماداً على البيانات الحالية فقط.

### الأعمال

- إعادة تصميم صفحة العملاء والطلبات.
- إزالة عناصر الدفع والفواتير من الواجهة.
- تحسين كروت أو قائمة العملاء.
- إعادة ترتيب مساحة عمل العميل.
- حذف أو إخفاء الأزرار غير العاملة.
- تحسين empty/loading/error states.
- إبقاء `Order` كما هو في قاعدة البيانات لكن عرضه في الواجهة باسم طلب أو مشروع.

### الملفات المتوقعة

- `src/components/Admin/AdminDashboard.jsx`
- `src/components/Admin/ClientWorkspace.jsx`
- ربما إنشاء مكونات جديدة داخل:
  - `src/components/Admin/clients`
  - `src/components/Admin/workspace`

### مستوى المخاطرة

منخفض.

### لا يتم في هذه المرحلة

- لا تغيير Prisma schema.
- لا رفع ملفات حقيقي.
- لا صفحة preview عامة.
- لا تغيير backend كبير.

## Phase 2 - تحسين تدفق البيانات

### الهدف

جعل بيانات العملاء والطلبات أوضح وأخف.

### الأعمال

- فصل client summary عن client details.
- تقليل البيانات التي ترجع للوحة العامة.
- تحسين endpoints للعميل الواحد.
- تحسين حفظ brand kit.
- إضافة validation أقوى للحقول.

### مستوى المخاطرة

متوسط.

## Phase 3 - نظام تسليم حقيقي

### الهدف

تحويل Deliverables إلى نظام حقيقي لتسليم thumbnails.

### الأعمال

- اختيار storage خارجي:
  - Cloudinary
  - Supabase Storage
  - Cloudflare R2
- رفع الصور وحفظ روابطها.
- ربط التسليم بطلب معين.
- عرض النسخ في مساحة العميل.
- نسخ رابط preview حقيقي.

### ملاحظة مهمة

Render filesystem مؤقت، لذلك لا يجب الاعتماد على مجلد `uploads` لتسليم thumbnails.

### مستوى المخاطرة

مرتفع نسبياً لأنه يمس storage وbackend وربما database schema.

## Phase 4 - نظام طلبات احترافي

### الهدف

إنشاء workflow كامل للطلبات.

### الأعمال

- طلب يحتوي brief واضح.
- حالة الطلب.
- revision notes.
- روابط فيديو أو ملفات مرجعية.
- اختيار نسخة معتمدة.
- timeline لكل طلب.

### مستوى المخاطرة

متوسط إلى مرتفع، حسب تغييرات قاعدة البيانات.

## Phase 5 - صفحة Preview للعميل

### الهدف

إنشاء رابط احترافي يمكن إرساله للعميل لمراجعة النسخ.

### الخصائص

- صفحة بدون دخول.
- رابط خاص أو token.
- عرض النسخ.
- تحميل النسخة.
- اختيار أو إرسال ملاحظة.

### مستوى المخاطرة

مرتفع نسبياً لأنه يحتاج حماية رابط ومشاركة خارجية.

## أول خطوة تنفيذ موصى بها

ابدأ بـ Phase 1 فقط.

السبب:

- تعطي فرق كبير في الشكل والاستخدام.
- لا تكسر الإنتاج.
- لا تحتاج database migration.
- تساعدنا نثبت تجربة العمل قبل بناء storage وتسليم حقيقي.
